## 智能画布生成链路核心规则

### 修改生图逻辑前必须备份

每次修改参考图路由、人物相似度、产品约束、提示词压缩、相机覆盖、API 请求参数、
生成任务状态或空白节点清理之前，必须先运行：

```powershell
powershell -ExecutionPolicy Bypass -File .\tools\backup-generation-logic.ps1 -Reason "本次修改说明"
```

备份保存在 `backups/generation-logic/时间戳/`，包含 `main.py`、
`static/js/smart-canvas.js`、`static/smart-canvas.html`、项目手册和 SHA256 清单。
禁止覆盖或删除已有备份。

### 1. 输出节点状态必须完整闭环

生成输出节点只能处于以下几类状态之一：

- 等待中：`pending > 0` 或 `pendingTasks.length > 0` 或 `jimengPending`。
- 成功：`images` 中至少有一个真实 `url`，并且 `runFinishedAt` 有值。
- 失败/取消：必须清掉 `pending`、`pendingTasks`、`running`、`queued` 和运行元数据。

禁止留下这种“幽灵输出节点”：

- 没有图片；
- 没有 pending task；
- 不是 running；
- 没有 `runFinishedAt`；
- 但仍保留 `runPrompt`、`runModelPrompt`、`sourceNodeId`、`runAt` 或 `runStartedAt`。

这类节点会在画布上表现为空白卡片、错误计时、连接线残留。前端已通过 `filterStaleEmptyRunPlaceholders` 在加载时清理，但新代码仍要避免制造它们。

异步任务恢复/续查也必须即时收口：

- 如果任务失败但返回了可恢复的上游任务 ID，保留 `pendingTasks[].failed/recoverTaskId`，允许用户手动“查询结果”。
- 如果任务失败且没有任何图片、没有可恢复任务 ID，必须当场调用失败清理逻辑；自动分支输出节点应直接移除并写入 tombstone，避免被协作同步再次复活。
- `resumeSmartPendingTasks()` 是后台恢复流程，不能产生未处理的 Promise 异常；失败应被捕获并保存清理后的画布状态。
- `runGeneration()` 必须先确认存在选中节点，再构建 prompt 请求，避免空选择状态抛出前端异常。

### 2. `sourceNodeId` 不能指向自己

`sourceNodeId` 只表示“这个输出来自哪个上游源节点”。如果目标节点就是源节点本身，不能写入 `sourceNodeId`。

正确逻辑：

- 分支输出节点：`sourceNodeId = 源节点 id`。
- 原节点直接生成：不写 `sourceNodeId`。
- 输出节点失败并且无图片：调用失败清理逻辑，删除运行元数据。

### 3. 多图/二轮生成注意事项

当底部数量为 `2张`、`3张` 等多任务时：

- `pending` 初始值应等于任务数；
- 每个任务成功后只递减一次；
- 所有任务完成后必须删除 `pendingTasks`，写入 `runFinishedAt`；
- 若任务失败且节点没有图片，应清理节点或删除分支输出节点；
- 不允许把失败空节点继续作为下一轮可复用输出。

`reusableAutoOutputForSource` 只能复用已经完整完成且可显示的输出节点，不能复用空运行占位。

## 参考图与产品约束规范

### 1. 参考图角色

- 产品图：作为严格 SKU 事实参考，允许上传，优先约束产品结构、材质、颜色、标签位置、比例和细节。
- 人像/风格图：默认弱参考。中低权重时尽量转为文字风格参考，不上传原图，避免人物脸完全复制。
- 模板库匹配图：只用于提示词/风格描述，任何权重和场景下都不作为 API 图片参考上传。

### 1.1 人物/肢体出现权限

生成链路默认是“产品/物体优先”，不能因为案例库、风格库、旧提示词模板里出现 `model`、`hand placement`、`skin treatment`、`pose` 等词，就自动生成真人、手、脸或身体。

只有满足以下任一条件，才允许出现人物、手、脸、皮肤、肢体或人体剪影：

- 真实上传/连接的风格参考图经分析明确包含人物或肢体；
- 用户当前提示词、提示词节点或上游文本中明确要求人物、模特、手持、手拿、肢体、佩戴、真人场景等；
- 用户明确要求复刻/保留有人物的生活方式场景。

否则必须追加 `NO HUMAN SUBJECT LOCK`：

- 不出现 person / people / human / model / portrait / face / skin / hair / hand / fingers / arms / body / silhouette / mannequin；
- 产品可通过支架、台面、道具、机械结构、光影、粒子、环境来承托；
- 纯产品场景不能为了“手持关系”自动加入手。

### 2. 人物相似度

参考权重控制的是“风格/构图/人物类型相似度”，不是默认复制同一个人。

- 低权重：新人物、新姿势、新构图。
- 中权重：同商业风格，但人脸必须明显不同。
- 高权重：可更接近参考，但仍不要默认完全同脸，除非用户明确要求锁定人物。

### 2.1 参考图上传尺寸与性能

- 产品真值图最长边最多 `1536`，保留结构、材质和标签细节。
- 普通人物/构图/风格参考在权重 `<85` 时最长边最多 `768`。
- 高权重人物/构图/风格参考在权重 `>=85` 时最长边最多 `1024`。
- 模板案例图永不上传，后端必须无条件过滤 `case_style_reference`，防止案例图片意外进入请求。
- OpenAI multipart 编辑请求需要缩放时，只生成 `sized_ref_` 临时副本；不得修改或覆盖画布中的原始文件。

### 3. 产品手持关系

如果用户要求产品被拿着，提示词必须强调真实物理接触：

- 产品必须在手中；
- 至少两个手指或掌心与产品产生遮挡/接触；
- 不允许产品漂浮在手边；
- 手、产品、阴影、遮挡关系要合理。

## 尺寸/比例约束规范

底部尺寸选择是最高优先级之一。

- 用户选择 `9:16 · 2K` 时，最终输出必须按目标画布比例生成/保存。
- 参考图比例不能覆盖用户选择的输出比例。
- 老提示词里的 `Preserve its aspect ratio`、`strict composition` 等内容必须在运行前被弱化或覆盖。
- 前端会追加 `OUTPUT CANVAS LOCK`，告诉模型按用户选择的画布生成。
- 后端保存图片时会做兜底比例校正：如果上游返回比例不对，会本地转成请求尺寸。优先扩展/模糊背景，不硬裁产品和手部。
- `auto` 模式不强制比例。
- 异步生图任务、手动查询上游任务结果、同步返回结果都必须把请求 `size` 传给保存函数；否则第二轮/续查结果会绕过比例兜底。

## 提示词长度与压缩

提示词太长时，压缩必须保留这些高优先级段落：

- 用户最新修改要求；
- 产品事实/产品替换规则；
- `OUTPUT CANVAS LOCK`；
- 人物身份控制；
- 手持产品交互规则；
- 上传图片角色映射；
- 最终产品检查。

模板库、案例库、风格解释可以压缩；产品事实和输出画布约束不能被压掉。

## GPT-Image2 风格库集成规范

项目内置风格库目录：

```text
style_libraries/gpt-image-2/
```

后端解析顺序：

1. 如果设置了环境变量 `STYLE_LIBRARY_DIR`，优先使用该目录。
2. 否则优先使用项目内 `style_libraries/gpt-image-2/`。
3. 如果项目内库不存在，再回退到本机 Codex skill：`~/.codex/skills/gpt-image-2-style-library`。

这样上传到 GitHub 后，另一台电脑只要拉取本项目，就能继续使用画布里的“匹配提示词 / 风格案例检索”能力，不依赖当前电脑的 Codex skill 安装状态。

纳入仓库的内容只包含运行所需的检索脚本和参考库：

- `scripts/search_all.py`
- `references/*.json`
- `references/*.md`

不要提交风格库的日志、daily update state/cache、Codex agent 配置、本机私有 skill 目录、API Key 或任何用户输出文件。

### 自动更新

无限画布服务启动时会后台检查项目内风格库是否今天已更新：

- 如果 `style_libraries/gpt-image-2/references/daily-update-state.json` 的 `last_check_date` 是今天，则跳过；
- 如果今天未更新，则后台运行 `style_libraries/gpt-image-2/scripts/update_daily_sources.py`；
- 更新日志写入 `style_libraries/gpt-image-2/references/daily-update.log`；
- 锁文件为 `style_libraries/gpt-image-2/references/.daily-update.lock`；
- 网络失败时继续使用本地缓存，不影响画布启动和生图；
- 可用环境变量 `STYLE_LIBRARY_AUTO_UPDATE=0` 关闭；
- 可用 `STYLE_LIBRARY_AUTO_UPDATE_TIMEOUT=秒数` 调整最长等待时间。

自动更新只操作项目内 `style_libraries/gpt-image-2/`，不会写入 `C:\Users\pc\.codex\skills\gpt-image-2-style-library`，因此本机 Codex skill 不受影响。
